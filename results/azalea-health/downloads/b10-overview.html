170.315 (b)(10) requires the implementation of Electronic Health Information (EHI) export functionality. This encompasses the ability to generate export files for individual patients and entire patient populations in a computable format. Azalea Health provides comprehensive support for this requirement through each of its platform's Bulk Data Export feature. File Format Bulk Data Export files are formatted in NDJSON (New line delimited JSON), a modified version of the JSON format designed for efficient bulk data transfer. NDJSON follows a simplified JSON structure where resources are serialized without whitespace and separated by a newline pair (ASCII characters 13 and 10). This format aids in streamlining processing by ensuring that each NDJSON document contains only resources of a single type. Each line within the document represents a resource of that specific type. A single file can accommodate multiple resources for multiple patients with multiple records, although the resource type is the same across all entries per file. Each NDJSON file will have a file naming convention of {resource}_{identifier}.ndjson where {resource} represents a FHIR resource type, i.e., Patient or Encounter and {identifier} is a randomized unique file identifier. Example: Patient_24r0sadf098230.ndjson. Within each NDJSON file, only the resources of the named FHIR resource type will be included. Resources Included When exporting resources, all data from within the Patient compartment, as defined by HL7 FHIR R4, will be included. The following additional, supporting administrative resources are also included: Location, Practitioner. Unstructured Data EHI also covers unstructured documents that are associated with a patient as well, such as images, documents, or medical notes. Binary Resources are used to represent these kinds of unstructured data and all Binary resources linked to the relevant Patient are included in the export alongsize all of the Patient compartment resources. The unstructured data is in every case accessible as a base 64 encoded string within the Binary.data JSON property. In some cases unstructured data may be accessible directly inside the DocumentReference resource (for example, exported clinical notes). The unstructured data in these cases is also accessible as a base 64 encoded string within the DocumentReference.content.attachemnt.data JSON property. In all other cases the DocumentReference.content.attachemnt.url JSON property will point to a Binary resource which may be found within the exported Binary resources.
Single Patient EHI Export - 170.315(b)(10)(i)
To meet the requirements of 170.315(b)(10)(i), each EHR platform provides an in app user interface for requesting a single patient export. Once the request is submitted the single patient export will be run in the background and complete with no additional user intervention. A single patient can be also be exported via the Bulk Data Export API by adding a _typeFilter query parameter to the all patient export request to filter the Patient resource by a single _id: 
GET [base]/Patient/$export?_typeFilter=Patient?_id=1234
Accept: application/fhir+json
Prefer: respond-async
Patient Population EHI Export - 170.315(b)(10)(ii)
To meet the requirements of 170.315(b)(10)(ii), users requiring a patient population export must contact support to initiate the request, as the volume of data covered could be quite large. Our staff will work with you on the best method for delivering the exported files to you in an efficient manner. There is no charge for this export type. Alternatively, a patient population export can be also be acheived via the Bulk Data Export API through either the All Patient export or the Group of Patients export endpoints.
